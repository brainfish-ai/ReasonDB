ReasonDB isn't an incremental improvement—it's a fundamental rethinking of how databases should work in the AI era. This page explains the core concepts that make intelligent document retrieval possible.

## The Key Insight

**Documents have structure. That structure has meaning. Losing it is losing intelligence.**

When you read a legal contract, you don't scan every word hoping to stumble upon the termination clause. You look at the table of contents, navigate to the relevant section, and drill down. ReasonDB gives AI the same capability.

<Info>
**Hierarchical Reasoning Retrieval (HRR)** is ReasonDB's core innovation: preserving document structure and using LLMs to navigate it intelligently, rather than embedding everything into a flat vector space.
</Info>

## Documents Become Trees

When you ingest a document, ReasonDB transforms it into a navigable tree:

<Mermaid>
{`graph TD

    Root --> D["4. Termination"]

    C --> C3["3.3 Late Penalties"]

**Why trees?**
- They mirror how documents are actually organized
- Each level provides increasing specificity
- Navigation is intuitive for both humans and AI
- Context is preserved at every level

## Nodes: The Building Blocks

Every piece of content in ReasonDB lives in a **node**. Nodes are the atomic units of knowledge.

| Property | Purpose | Example |
|----------|---------|---------|
| `id` | Unique identifier | `node_7f3a9b2c` |
| `title` | Section heading | "3.2 Payment Schedule" |
| `content` | Actual text | "Payments are due within 30 days..." |
| `summary` | LLM-generated navigation aid | "Specifies payment timing, grace periods, and invoicing requirements" |
| `depth` | Level in tree (0 = root) | `2` |
| `children` | Child node IDs | `["node_abc", "node_def"]` |
| `parent` | Parent node ID | `"node_xyz"` |

### Leaf vs Branch Nodes

<CardGroup cols={2}>
  <Card title="Branch Nodes" icon="folder">
    Have children. Contain section overviews and summaries. Used for navigation decisions.
  </Card>
  <Card title="Leaf Nodes" icon="file">
    No children. Contain the actual detailed content. Where answers are extracted.
  </Card>
</CardGroup>

**The summary is crucial.** When the LLM navigates the tree, it reads summaries to decide which branches are worth exploring. A good summary captures what questions this section can answer.

## LLM-Guided Traversal

This is where ReasonDB diverges from every other approach. Instead of retrieving then reasoning, ReasonDB **reasons to retrieve**.

### The Traversal Process

<Mermaid>
{`sequenceDiagram
    participant User
    participant ReasonDB
    participant LLM
    participant Tree
    
    User->>ReasonDB: "What are the late payment penalties?"
    
    ReasonDB->>Tree: Get root children summaries
    ReasonDB->>LLM: Which sections might answer this?
    LLM-->>ReasonDB: "Payment Terms" (0.92), "Termination" (0.31)
    
    Note over ReasonDB: Prune low-confidence branches
    
    ReasonDB->>Tree: Get "Payment Terms" children
    ReasonDB->>LLM: Which subsection is most relevant?
    LLM-->>ReasonDB: "Late Penalties" (0.97)
    
    ReasonDB->>Tree: Get leaf content
    ReasonDB->>LLM: Extract answer from content
    LLM-->>ReasonDB: "Late payments incur 1.5% monthly interest..."
    
    ReasonDB-->>User: Answer + confidence + path`}
</Mermaid>

### Why This Works Better

| Vector Search | ReasonDB Traversal |
|---------------|-------------------|
| Embeds query and chunks into same space | LLM reasons about relevance in natural language |
| "Similar" vectors might be irrelevant | LLM understands semantic intent |
| Loses document structure | Preserves and leverages structure |
| Same approach for all queries | Adapts depth based on query complexity |
| Black box retrieval | Explainable navigation path |

## Beam Search: Parallel Exploration

Real questions sometimes have answers in multiple places, or the best path isn't obvious upfront. ReasonDB uses **beam search** to explore multiple promising branches simultaneously.

```
Query: "What happens if I miss a payment?"

Beam Width: 3

Level 1 - Evaluate all sections:
  → "Payment Terms" (0.88) ✓ Keep
  → "Termination" (0.72) ✓ Keep  
  → "Definitions" (0.15) ✗ Prune
  → "Services" (0.08) ✗ Prune

Level 2 - Explore top 3 branches in parallel:
  → Payment Terms → Late Penalties (0.94) ✓
  → Payment Terms → Payment Schedule (0.45) ✓
  → Termination → Termination for Cause (0.68) ✓

Level 3 - Continue to leaves, extract answers
```

### Tuning Beam Search

| Parameter | Default | Effect |
|-----------|---------|--------|
| `beam_width` | 3 | More branches = more thorough, slower |
| `max_depth` | 10 | Deeper traversal for complex documents |
| `min_confidence` | 0.3 | Prune branches below this threshold |

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{
    "query": "What are all the ways the contract can end?",
    "beam_width": 5,
    "min_confidence": 0.25
  }'
```

<Tip>
Increase `beam_width` for exploratory questions where answers might be scattered. Decrease for specific factual queries.
</Tip>

## Tables: Organizing Knowledge

**Tables** are collections of documents. Think of them as namespaces or folders for your knowledge base.

```bash
# Create topical collections
curl -X POST http://localhost:4444/v1/tables \
  -d '{"name": "Legal Contracts", "metadata": {"department": "legal"}}'

curl -X POST http://localhost:4444/v1/tables \
  -d '{"name": "Technical Docs", "metadata": {"department": "engineering"}}'
```

**Why tables matter:**
- Scope searches to relevant collections
- Different retention and access policies
- Track document counts and usage
- Organize by department, project, or topic

## Search vs Query

ReasonDB offers two complementary interfaces:

### `/v1/search` — LLM-Guided Retrieval

For natural language questions requiring reasoning:

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "What happens to intellectual property if the contract is terminated?"}'
```

**Best for:** Complex questions, answer extraction, reasoning across sections

### `/v1/query` — RQL Structured Queries

For filtering, aggregation, and hybrid search:

```bash
curl -X POST http://localhost:4444/v1/query \
  -d '{
    "query": "SELECT * FROM \"Legal Contracts\" WHERE metadata.status = '\''active'\'' SEARCH '\''termination'\'' LIMIT 10"
  }'
```

**Best for:** Filtering by metadata, keyword search, combining filters with reasoning

<Info>
RQL's `REASON` clause lets you combine structured queries with LLM reasoning—filter first, then reason over the results.
</Info>

## The Ingestion Pipeline

Understanding how documents become trees helps you optimize for your use case:

<Mermaid>
{`flowchart TB
    A[📄 PDF / Markdown / HTML / Text] --> B[Extract Text]
    B --> C[Chunk by Structure]
    C --> D[Build Hierarchical Tree]
    D --> E[Generate LLM Summaries]
    E --> F[(Knowledge Base)]`}
</Mermaid>

1. **Extract**: Parse document format, preserve structural hints
2. **Chunk**: Split by headings, paragraphs, semantic boundaries
3. **Build Tree**: Create hierarchy from document outline
4. **Summarize**: Generate LLM summaries bottom-up (leaves first, then parents)
5. **Index**: Store with BM25 text search and tree structure

<Warning>
**Summary quality is critical.** Poor summaries lead to poor navigation. ReasonDB uses your configured LLM to generate summaries—better LLM = better summaries = better search.
</Warning>

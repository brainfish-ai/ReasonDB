ReasonDB search is fundamentally different. Instead of returning "similar" chunks and hoping one is useful, the LLM actively navigates your document structure to find exactly what you need—then extracts a precise answer.

## The Basics

Ask a question. Get an answer.

```bash
curl -X POST http://localhost:4444/v1/search \
  -H "Content-Type: application/json" \
  -d '{"query": "What are the late payment penalties?"}'
```

```json
{
  "results": [{
    "node_id": "node_payment_3",
    "document_id": "doc_contract",
    "path": ["Service Agreement", "Payment Terms", "Late Penalties"],
    "content": "Late payments will incur interest at 1.5% per month...",
    "answer": "Late payments incur 1.5% monthly interest, compounded. After 60 days, service may be suspended.",
    "confidence": 0.94
  }],
  "stats": {
    "nodes_visited": 5,
    "nodes_pruned": 8,
    "llm_calls": 4,
    "total_time_ms": 1850
  }
}
```

**What happened:**
1. LLM read top-level summaries → identified "Payment Terms" as relevant
2. Drilled into subsections → found "Late Penalties"
3. Extracted and synthesized the answer
4. Returned the path taken and confidence score

## Search Parameters

Fine-tune search behavior for your use case:

| Parameter | Default | Purpose |
|-----------|---------|---------|
| `query` | *required* | Your natural language question |
| `beam_width` | 3 | Branches to explore in parallel |
| `max_depth` | 10 | How deep to traverse |
| `min_confidence` | 0.3 | Prune threshold |
| `table_id` | all | Restrict to specific table |
| `document_id` | all | Search single document |
| `limit` | 10 | Max results to return |

## Scoping Your Search

### Search a Specific Table

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{
    "query": "confidentiality obligations",
    "table_id": "tbl_legal_contracts"
  }'
```

### Search a Single Document

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{
    "query": "What is the notice period?",
    "document_id": "doc_employment_agreement"
  }'
```

### Filter by Metadata

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{
    "query": "liability limits",
    "metadata": {
      "contract_type": "service_agreement",
      "status": "active"
    }
  }'
```

### Filter by Tags

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{
    "query": "intellectual property rights",
    "tags": ["nda", "confidential"]
  }'
```

## Tuning for Quality vs Speed

### Beam Width: Breadth of Exploration

**Low beam width (1-2)**: Fast, focused. Best for specific factual questions.

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "What is the contract effective date?", "beam_width": 1}'
```

**High beam width (4-5)**: Thorough, slower. Best for complex questions with multiple relevant sections.

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "What are all the ways this contract can be terminated?", "beam_width": 5}'
```

### Min Confidence: Pruning Aggressiveness

**High threshold (0.5+)**: Aggressive pruning, faster, might miss edge cases.

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "...", "min_confidence": 0.5}'
```

**Low threshold (0.2)**: Explores more branches, slower, more comprehensive.

```bash
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "...", "min_confidence": 0.2}'
```

### Max Depth: How Deep to Go

```bash
# High-level overview (fast)
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "...", "max_depth": 3}'

# Detailed deep search (thorough)
curl -X POST http://localhost:4444/v1/search \
  -d '{"query": "...", "max_depth": 15}'
```

## Understanding Results

### The Path: How ReasonDB Got There

Every result includes the navigation path—this is explainability built in:

```json
{
  "path": [
    "Service Agreement",
    "Section 5: Payment Terms", 
    "5.2 Late Payment Penalties"
  ]
}
```

You know exactly which sections the LLM traversed. This is invaluable for:
- Debugging unexpected results
- Building trust in AI-generated answers
- Providing citations in downstream applications

### Confidence Scores

| Score | Meaning |
|-------|---------|
| **0.9+** | High confidence—directly answers the question |
| **0.7-0.9** | Strong match—likely what you're looking for |
| **0.5-0.7** | Moderate—related but may not be exact |
| **0.3-0.5** | Low—tangentially related |
| **< 0.3** | Pruned—not returned |

### The Answer vs The Content

```json
{
  "content": "Section 5.2: Late payments shall incur interest at a rate of 1.5% per calendar month, calculated from the due date. Interest shall compound monthly. After 60 days of non-payment, Provider reserves the right to suspend services...",
  
  "answer": "Late payments incur 1.5% monthly interest (compounding). After 60 days, services may be suspended."
}
```

- **content**: The raw text from the document
- **answer**: LLM-synthesized response to your specific question

## Search vs RQL Queries

| Need | Use |
|------|-----|
| Natural language question | `POST /v1/search` |
| Filter + keyword search | `POST /v1/query` with `SEARCH` |
| Filter + LLM reasoning | `POST /v1/query` with `REASON` |
| Aggregations | `POST /v1/query` |

### Combining Filters with Reasoning

For complex scenarios, use RQL with the REASON clause:

```bash
curl -X POST http://localhost:4444/v1/query \
  -d '{
    "query": "SELECT * FROM \"Legal Contracts\" WHERE metadata.status = '\''active'\'' AND metadata.value_usd > 100000 REASON '\''What are the termination conditions?'\'' LIMIT 5"
  }'
```

This filters first (fast), then reasons over the filtered set (thorough).
